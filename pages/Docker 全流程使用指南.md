# Docker 全流程使用指南
#Docker #容器化 #DevOps #项目部署 #运维
- ## 一、Docker 核心适用场景（什么情况下应该使用Docker）
	- 核心适用场景
		- 项目环境一致性保障：解决「本地能跑、线上不能跑」的环境差异问题，覆盖开发、测试、生产全流程[[环境一致性]]
		- 微服务架构拆分与部署：为每个[[微服务]]实例提供独立运行环境，实现服务隔离与快速扩缩容
		- 多版本/多环境并行测试：快速启动多套隔离的应用实例，无需担心环境冲突
		- 跨平台应用交付：一次构建、到处运行，支持X86/ARM等多架构部署，适配Linux/Windows/macOS
		- 持续集成/持续部署(CI/CD)流水线：与[[CI/CD]]工具无缝集成，实现自动化构建、测试、发布
		- 资源隔离与高效利用：单主机可运行多个隔离容器，相比虚拟机资源占用更低、启动速度更快
	- 不推荐使用的边界场景
		- 极致性能要求的底层应用（如数据库集群核心节点、高频交易系统），容器虚拟化存在极微性能损耗
		- 强依赖宿主机硬件/内核特性的专用程序
		- 超小型单体应用，无环境依赖、无多环境部署需求，使用Docker会增加不必要的复杂度
- ## 二、Docker 容器功能拆分决策逻辑（如何决策每个Docker的功能）
	- 核心决策原则：[[单容器单职责原则]]（一个容器只运行一个主进程/一个核心功能）
	- 功能拆分核心维度
		- 业务边界拆分：按业务领域/微服务模块拆分，如用户服务、订单服务、支付服务各独立一个容器
		- 技术栈拆分：不同技术栈/运行环境的服务必须拆分，如Java后端、Node.js前端、Python数据分析服务各独立容器
		- 生命周期拆分：启停频率、扩缩容策略不同的服务必须拆分，如高频访问的API服务、低频运行的定时任务服务拆分
		- 状态拆分：有状态服务（如数据库、缓存）与无状态服务（如API服务）必须拆分，便于状态管理与扩缩容
		- 权限拆分：不同安全等级、不同权限需求的服务必须拆分，降低安全风险
	- 决策校验标准
		- 容器内主进程崩溃，容器即退出，符合容器生命周期与进程生命周期绑定的设计
		- 单个容器的镜像构建、更新、重启不影响其他容器的正常运行
		- 单个容器可独立进行扩缩容、版本回滚、监控告警
	- 禁止的反模式
		- 单容器内运行多个核心服务（如同时运行Nginx+Java+MySQL）
		- 把无状态业务与有状态数据存储放在同一个容器
		- 容器功能粒度过细，导致容器数量过多、运维复杂度指数级上升
- ## 三、Docker 镜像与容器创建全流程
	- 核心概念
		- [[Docker 镜像]]：容器的只读模板，包含应用运行所需的代码、运行环境、依赖、配置文件
		- [[Docker 容器]]：镜像的运行实例，是一个独立的可运行环境
		- [[Dockerfile]]：用于构建镜像的文本文件，包含一系列构建指令
	- 方式一：通过[[Dockerfile]]构建自定义镜像（推荐）
		- 基础步骤
			- 1. 新建空白目录作为构建上下文，创建`Dockerfile`文件
			- 2. 编写Dockerfile核心指令
				- `FROM`：指定基础镜像（如`FROM openjdk:17-jdk-slim`、`FROM nginx:alpine`），必须作为首行
				- `WORKDIR`：指定容器内工作目录，避免绝对路径混乱
				- `COPY/ADD`：将本地文件复制到镜像内，`COPY`优先用于普通文件，`ADD`支持自动解压压缩包
				- `RUN`：镜像构建阶段执行的命令，用于安装依赖、编译代码
				- `ENV`：设置环境变量，用于配置应用运行参数
				- `EXPOSE`：声明容器要暴露的端口，仅作为文档说明，不自动开启端口映射
				- `CMD/ENTRYPOINT`：容器启动时执行的命令，`CMD`可被运行时参数覆盖，`ENTRYPOINT`用于固定启动入口
			- 3. 执行构建命令，生成镜像
				- 核心命令：`docker build -t 镜像名:版本号 .`
				- 示例：`docker build -t my-java-app:v1.0 .`
				- 关键参数：`-f` 指定非默认名称的Dockerfile，`--platform` 指定镜像架构
			- 4. 验证镜像构建结果：`docker images` 查看已构建的镜像列表
		- 最佳实践
			- 使用官方轻量基础镜像（alpine版本），减小镜像体积
			- 优化[[镜像分层]]，合并RUN指令，使用`.dockerignore`排除无关文件
			- 多阶段构建，分离编译环境与运行环境，剔除构建依赖
	- 方式二：从已有运行容器创建镜像
		- 适用场景：临时修改容器内容后，固化为新镜像
		- 核心命令：`docker commit -m "提交说明" -a "作者" 容器ID/容器名 新镜像名:版本号`
		- 注意：不推荐用于生产环境，无法追溯构建过程，不符合DevOps规范
	- 方式三：从官方/公共镜像仓库拉取现成镜像
		- 核心命令：`docker pull 镜像名:版本号`
		- 示例：`docker pull mysql:8.0`、`docker pull redis:7-alpine`
		- 推荐仓库：[[Docker Hub]]、阿里云容器镜像服务、腾讯云容器镜像服务
- ## 四、Docker 容器运行与生命周期管理
	- 核心命令：[[docker run]]（创建并启动容器）
		- 基础语法：`docker run [可选参数] 镜像名:版本号 [启动命令]`
		- 必选/高频参数详解
			- `-d`：后台守护进程模式运行容器，返回容器ID
			- `--name`：为容器指定唯一名称，便于管理，避免使用随机名称
			- `-p 宿主机端口:容器端口`：端口映射，将容器内端口暴露到宿主机，实现外部访问
				- 示例：`-p 8080:80` 将容器内80端口映射到宿主机8080端口
			- `-v 宿主机路径:容器内路径`：[[数据卷]]挂载，实现数据持久化，容器删除后数据不丢失
				- 示例：`-v /host/mysql/data:/var/lib/mysql` 挂载MySQL数据目录
			- `-e 环境变量名=值`：传入环境变量，用于配置应用，如数据库账号密码
			- `--restart=always`：设置容器开机自启，宿主机重启后自动启动容器
			- `--network`：指定容器所属的网络，用于容器间通信
		- 完整运行示例
			- 启动Nginx容器：`docker run -d --name my-nginx -p 80:80 -v /host/nginx/html:/usr/share/nginx/html --restart=always nginx:alpine`
			- 启动MySQL容器：`docker run -d --name my-mysql -p 3306:3306 -v /host/mysql/data:/var/lib/mysql -e MYSQL_ROOT_PASSWORD=123456 --restart=always mysql:8.0`
	- 容器生命周期全流程管理命令
		- 查看容器列表
			- `docker ps`：查看正在运行的容器
			- `docker ps -a`：查看所有容器（包括已停止的）
		- 容器启停操作
			- `docker start 容器ID/容器名`：启动已停止的容器
			- `docker stop 容器ID/容器名`：优雅停止运行中的容器
			- `docker restart 容器ID/容器名`：重启容器
			- `docker kill 容器ID/容器名`：强制终止容器
		- 容器日志与调试
			- `docker logs 容器ID/容器名`：查看容器运行日志
				- 高频参数：`-f` 实时跟踪日志，`--tail 100` 查看最后100行日志
			- `docker exec -it 容器ID/容器名 /bin/bash`：进入运行中的容器内部，执行交互命令
				- 说明：`-i` 保持输入流开启，`-t` 分配伪终端，alpine镜像需使用`/bin/sh`
		- 容器删除操作
			- `docker rm 容器ID/容器名`：删除已停止的容器
			- `docker rm -f 容器ID/容器名`：强制删除运行中的容器
			- 批量清理：`docker container prune` 清理所有已停止的容器
	- 运行最佳实践
		- 容器内应用必须前台运行，否则容器启动后会立即退出
		- 所有持久化数据必须通过[[数据卷]]挂载到宿主机，禁止将数据存储在容器内
		- 禁止使用`latest`标签部署生产环境，必须指定固定版本号，避免版本不一致问题
		- 配置健康检查`--health-cmd`，监控容器内应用运行状态，实现自动重启故障容器
- ## 五、Docker 容器间通信完整方案
	- 核心前提：[[Docker 网络模式]]，Docker默认提供5种网络驱动
		- `bridge`：默认桥接网络，同一主机内的容器可通过此网络通信，默认隔离宿主机网络
		- `host`：共享宿主机网络命名空间，容器直接使用宿主机的IP和端口，无网络隔离
		- `container`：共享指定容器的网络命名空间，多个容器共用同一个网络栈
		- `none`：无网络模式，容器关闭所有网络接口，仅本地回环
		- `overlay`：跨主机网络模式，用于Swarm集群或K8s集群，实现多主机容器通信
	- 单机容器间通信方案（同一宿主机）
		- 方案1：自定义bridge网络（**生产环境推荐**）
			- 核心优势：自带DNS解析，同一网络内的容器可直接通过容器名/别名互相访问，无需固定IP
			- 操作步骤
				- 1. 创建自定义bridge网络：`docker network create my-custom-network`
				- 2. 启动容器时加入该网络，通过`--network`指定
					- 示例1：`docker run -d --name my-mysql --network my-custom-network -e MYSQL_ROOT_PASSWORD=123456 mysql:8.0`
					- 示例2：`docker run -d --name my-java-app --network my-custom-network my-java-app:v1.0`
				- 3. 容器间直接通过容器名通信：Java应用内可直接通过`my-mysql:3306`访问MySQL数据库，无需端口映射
			- 补充操作
				- 为运行中的容器添加网络：`docker network connect 网络名 容器名`
				- 查看网络内的容器：`docker network inspect 网络名`
				- 断开容器网络：`docker network disconnect 网络名 容器名`
		- 方案2：host网络模式
			- 适用场景：性能要求极高的服务，无需网络隔离
			- 操作：启动容器时添加`--network host`参数，容器直接使用宿主机IP，端口直接占用宿主机端口，容器间通过`127.0.0.1:端口`直接通信
			- 注意：端口不能冲突，失去网络隔离性，生产环境慎用
		- 方案3：container网络模式
			- 适用场景：需要紧密耦合的多个容器，如Sidecar代理模式
			- 操作：启动容器时添加`--network container:已存在的容器名`参数，新容器与目标容器共享同一个网络栈，可通过`127.0.0.1`互相访问
		- 方案4：通过宿主机IP+映射端口通信（不推荐）
			- 操作：容器通过宿主机的IP+暴露的端口互相访问
			- 缺点：需要手动管理端口映射，网络绕路，性能损耗，无法动态适配容器IP变化
		- 方案5：--link 链接（已废弃，不推荐）
			- 说明：Docker已废弃该特性，仅兼容旧版本，无安全性，无法动态更新
	- 跨主机容器间通信方案（多台宿主机）
		- 方案1：[[overlay 网络]]（Docker原生方案）
			- 适用场景：Docker Swarm集群，开箱即用，原生支持
			- 核心特性：跨主机的分布式网络，同一overlay网络内的容器，无论在哪台主机，都可直接通过容器名互相访问，无需端口映射
		- 方案2：第三方CNI网络插件
			- 适用场景：Kubernetes集群或大规模容器部署
			- 主流方案：[[Calico]]、Flannel、Weave，提供网络策略、安全隔离、高性能转发
		- 方案3：端口映射+宿主机IP通信
			- 适用场景：少量跨主机容器，无集群环境
			- 操作：将容器端口映射到宿主机，通过宿主机的公网/内网IP+映射端口实现跨主机访问
			- 缺点：运维成本高，安全性低，不适合大规模部署
	- 通信安全最佳实践
		- 不同业务的容器组，创建独立的自定义网络，实现网络隔离
		- 仅暴露必要的端口到宿主机，非必要端口不做映射
		- 通过网络策略限制容器间的访问权限，仅允许白名单服务通信
		- 跨主机通信优先使用加密隧道，避免明文传输
- ## 六、Docker 迁移与跨环境部署全流程
	- 核心概念：[[镜像仓库]]，用于存储和分发Docker镜像，是跨环境部署的核心载体
	- 方案1：基于镜像仓库的标准化迁移部署（**生产环境推荐**）
		- 适用场景：开发、测试、生产多环境部署，CI/CD自动化流程，大规模集群部署
		- 全流程步骤
			- 1. 镜像构建与推送
				- 本地完成Dockerfile编写，构建镜像：`docker build -t 仓库地址/镜像名:版本号 .`
				- 登录镜像仓库：`docker login 仓库地址 -u 用户名 -p 密码`
				- 推送镜像到仓库：`docker push 仓库地址/镜像名:版本号`
				- 示例：`docker push registry.cn-hangzhou.aliyuncs.com/my-namespace/my-java-app:v1.0`
			- 2. 目标环境拉取与部署
				- 目标服务器登录镜像仓库：`docker login 仓库地址 -u 用户名 -p 密码`
				- 拉取对应版本镜像：`docker pull 仓库地址/镜像名:版本号`
				- 执行docker run命令启动容器，完成部署
			- 3. 版本管理与回滚
				- 每个发布版本使用唯一的版本号（如语义化版本、commit哈希）
				- 回滚时直接拉取历史版本镜像，重新启动容器即可
		- 推荐镜像仓库
			- 公共仓库：Docker Hub
			- 私有仓库：阿里云ACR、腾讯云TCR、Harbor（自建私有仓库）
	- 方案2：镜像离线迁移（save/load）
		- 适用场景：目标环境无外网，无法访问镜像仓库，内网离线部署
		- 全流程步骤
			- 1. 源环境将镜像导出为tar包
				- 命令：`docker save -o 导出文件名.tar 镜像名1:版本号 镜像名2:版本号`
				- 示例：`docker save -o my-app-images.tar my-java-app:v1.0 mysql:8.0 nginx:alpine`
				- 优势：导出完整镜像，包含所有分层、历史、标签，可保留完整镜像信息
			- 2. 将tar包传输到目标环境服务器（scp、U盘等方式）
			- 3. 目标环境导入tar包为本地镜像
				- 命令：`docker load -i 导出文件名.tar`
				- 示例：`docker load -i my-app-images.tar`
			- 4. 验证镜像导入结果：`docker images` 查看镜像，执行docker run启动容器
	- 方案3：容器文件系统迁移（export/import）
		- 适用场景：仅需迁移容器内的文件系统，无需保留镜像历史与分层，轻量化迁移
		- 全流程步骤
			- 1. 源环境将运行/停止的容器导出为tar包
				- 命令：`docker export -o 容器导出包.tar 容器ID/容器名`
			- 2. 目标环境导入tar包为新镜像
				- 命令：`docker import 容器导出包.tar 新镜像名:版本号`
		- 注意：export/import会丢失镜像历史、环境变量、启动入口等元数据，仅保留文件系统，不推荐用于常规镜像迁移
	- 方案4：基于[[Docker Compose]]的批量迁移部署（多容器项目推荐）
		- 适用场景：项目包含多个关联容器，需要批量部署、一键启停
		- 全流程步骤
			- 1. 编写`docker-compose.yml`文件，定义所有容器的配置、网络、数据卷、依赖关系
			- 2. 源环境构建镜像并推送到镜像仓库
			- 3. 将`docker-compose.yml`文件传输到目标环境
			- 4. 目标环境执行命令，一键启动所有容器
				- 命令：`docker compose up -d`
			- 5. 配套命令
				- 停止所有容器：`docker compose down`
				- 查看运行状态：`docker compose ps`
				- 查看日志：`docker compose logs -f`
	- 迁移部署最佳实践
		- 生产环境必须使用镜像仓库+固定版本号的部署方式，禁止离线tar包随意迁移，避免版本混乱
		- 所有配置文件、环境变量与镜像解耦，通过环境变量、配置挂载的方式注入目标环境
		- 持久化数据单独迁移，通过数据卷备份、恢复的方式同步到目标环境，禁止将数据打包到镜像内
		- 迁移前验证镜像在目标环境的架构兼容性（如X86与ARM架构差异）
		- 部署前做好端口、网络、权限的预检查，避免环境冲突导致部署失败
